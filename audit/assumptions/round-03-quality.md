# Phase 1 ŌĆö Round 03 ŌĆö Quality Discovery

## Repo Snapshot
Pulse mediator library targeting NET8/NET9/NET10; outbox pattern, idempotency, source-generated handler registration, multiple transport adapters (Kafka, RabbitMQ, Azure Service Bus, Dapr), multiple persistence providers (EF Core, ADO.NET for SQL Server/PostgreSQL/MySQL/SQLite). Round 03 focuses on the outbox write/read seam, idempotency key lifecycle, default-implementation pitfalls in `IMessageTransport`, generic type-name collisions, retry math, polymorphic serialization, and tracing/baggage propagation.

## Assumptions

Q33: `OutboxEventStore.StoreAsync` flushes the user's DbContext mid-business-transaction via `EntityFrameworkOutboxRepository.AddAsync` calling `SaveChangesAsync`, breaking "outbox + business writes" atomicity.
   Evidence: `src/NetEvolve.Pulse.EntityFramework/Outbox/EntityFrameworkOutboxRepository.cs:61-67` (`AddAsync` ŌåÆ `await _context.SaveChangesAsync(...)`). Called from `src/NetEvolve.Pulse/Outbox/OutboxEventStore.cs:92` via `IEventOutbox.StoreAsync`, which is invoked by `OutboxEventHandler<TEvent>` during `mediator.PublishAsync`. Any uncommitted business changes already tracked on the shared `DbContext` are also committed at that point; if the caller then throws or rolls back, the outbox row remains and is delivered without the business state it describes. Distinct from Q20: Q20 is about transaction-enlistment with ambient `TransactionScope`; this assumption is that the repository unconditionally flushes the *entire* shared DbContext, regardless of whether a transaction is open.
   Why it matters: The outbox is meant to guarantee at-least-once delivery only of state that was successfully committed. Premature flush of an unrelated UnitOfWork either over-publishes (event sent for state that was never persisted because of a downstream error) or, paired with `PulseMediator.PublishAsync` creating a *new* scope (see Q34), publishes to a different DbContext than the user's, making the outbox row outright unrelated to the user's transaction.
   Test idea: Register `AddPulse().AddOutbox().AddEntityFrameworkOutbox<TestContext>()`. In a handler: track an entity (`ctx.Items.Add(...)` without `SaveChanges`), then `await mediator.PublishAsync(evt)`. Assert that the tracked entity is now persisted (proving premature flush) and that the outbox table contains the event row. Then in a second test, after `PublishAsync`, throw before user calls `SaveChanges` ŌĆö assert outbox row still present (proving non-atomicity).

Q34: `PulseMediator.PublishAsync` creates a fresh DI scope (`CreateAsyncScope`) per event, so `OutboxEventHandler` resolves a *different* `IOutboxRepository`/`DbContext` than the caller's request scope, making "outbox row in same transaction as business writes" impossible by construction.
   Evidence: `src/NetEvolve.Pulse/Internals/PulseMediator.cs:86-100` (`var scope = _serviceProvider.CreateAsyncScope(); ... handlers = scope.ServiceProvider.GetServices<IEventHandler<TEvent>>()`). `OutboxEventHandler<TEvent>` is registered open-generic Scoped at `src/NetEvolve.Pulse/OutboxExtensions.cs:68`. Inside the fresh scope, `IOutboxRepository` resolves to a brand-new `EntityFrameworkOutboxRepository<TContext>` with a brand-new `TContext` (DI Scoped semantics). The user's `DbContext` from the ambient request scope is never touched. Distinct from Q01 (singleton consuming Scoped) and from Q20 (transaction-enlistment): Q34 is about the *scope mismatch between caller and the outbox handler*.
   Why it matters: Users who follow the canonical outbox guidance ("save business entities + outbox row in one transaction, then commit") will find that calling `mediator.PublishAsync` followed by `await dbContext.SaveChangesAsync()` writes the outbox row using a *different* DbContext, defeating atomicity. They cannot share a transaction without significant surgery.
   Test idea: Wrap the publish in `TransactionScope`. Inspect `_context` identity inside the OutboxRepository vs the user's injected `DbContext`. Assert they are different instances. Roll back the outer scope and assert the outbox row is still present in the database.

Q35: `OutboxEventHandler<TEvent>` is registered for *every* `IEvent` (open-generic TryAddEnumerable), so any in-process event without `IEventInProcess` is **both** dispatched to the user's local handlers *and* persisted to the outbox for re-delivery ŌĆö causing duplicate handler execution when the outbox transport loops back into the same process or fans out to a competing consumer in the same service.
   Evidence: `src/NetEvolve.Pulse/Outbox/OutboxEventHandler.cs:37-46` only short-circuits when `message is IEventInProcess { HandleInProcess: true }`. `src/NetEvolve.Pulse/OutboxExtensions.cs:68` registers it as `IEventHandler<>` via `TryAddEnumerable`. `src/NetEvolve.Pulse/Internals/PulseMediator.cs:90-99` invokes ALL `IEventHandler<TEvent>` instances. Nothing prevents a user handler and the outbox forwarder from both firing.
   Why it matters: Local side effects (e.g., `EmailNotificationHandler`) run during `PublishAsync`, then run again when the outbox processor publishes to the same broker the local service subscribes to. Users who add `AddOutbox()` to an already-working in-process design silently start double-executing every handler. The opt-in `IEventInProcess` marker is the wrong default (only marker-implementers are exempt); the safe default would be opt-in to outbox forwarding.
   Test idea: Register a simple `IEventHandler<DemoEvent>` that increments a counter, plus `AddOutbox()`. Publish one event. Assert (a) counter == 1, (b) outbox table contains one row. Then add a no-op transport that re-invokes the local mediator and assert the counter becomes 2 ŌĆö confirming the duplication path.

Q36: `NullMessageTransport` (the default registered by `AddOutbox`) silently completes every message without any operator warning, so a misconfigured deployment will look healthy while every outbox event is dropped on the floor.
   Evidence: `src/NetEvolve.Pulse/Outbox/NullMessageTransport.cs:17-21` returns `Task.CompletedTask`. `src/NetEvolve.Pulse/OutboxExtensions.cs:69` registers it via `TryAddSingleton<IMessageTransport, NullMessageTransport>()`. `IsHealthyAsync` is not overridden, so it defaults to `true` (interface default in `IMessageTransport.cs:107`). `OutboxProcessorHostedService.ProcessMessageAsync` then calls `MarkAsCompletedAsync` after a "successful" send, so the row leaves the queue as if delivered. There is no startup log, warning, or diagnostic when the default transport remains in place.
   Why it matters: Forgetting `UseMessageTransport<...>()` (e.g., when a docs example is copied) silently turns the outbox into a black hole. Compounded with Q33/Q35, business events would appear to be published, the outbox processor logs success, metrics counters increment, but no broker ever sees the data.
   Test idea: Register `AddPulse().AddOutbox().AddEntityFrameworkOutbox<TestContext>()` without calling `UseMessageTransport`. Publish an event. Wait for the processor to drain. Assert the outbox row's status is `Completed`. Assert that nothing was logged at Warning/Error level mentioning the null transport. Then assert no startup diagnostic was emitted naming the active transport.

Q37: `DefaultTopicNameResolver` uses `Type.Name`, which collides across closed generic instantiations (`MyEvent<Customer>` and `MyEvent<Order>` both yield `"MyEvent`1"`) and across distinct types in different namespaces.
   Evidence: `src/NetEvolve.Pulse/Outbox/DefaultTopicNameResolver.cs:13-18` returns `message.EventType.Name`. `Type.Name` for `MyApp.OrderCreated<Customer>` is `OrderCreated\`1` and for `MyApp.OrderCreated<Order>` is also `OrderCreated\`1`; for `A.Foo` and `B.Foo` both return `Foo`. There is no generic-arg disambiguation, no namespace prefix, and no sanitization of the `\`N` arity suffix.
   Why it matters: Subscribers that route by topic name will receive cross-pollinated payloads (a Customer payload arriving on a topic that was supposed to carry Order events). Broker-side schema validation (Kafka schema registry, Azure Service Bus subscriptions filtered by `MessageType`) may reject messages or, worse, route them to the wrong consumer group. Concrete-type collisions across namespaces (e.g. `Billing.OrderCreated` and `Shipping.OrderCreated`) silently merge into one topic. The `\`` backtick in `Type.Name` is also illegal in most broker topic naming rules (Kafka allows `[a-zA-Z0-9._-]`).
   Test idea: Create two events `MyEvent<Customer>` and `MyEvent<Order>` registered as separate types. Call `new DefaultTopicNameResolver().Resolve(...)` on both. Assert the resolved strings are equal (collision). Then assert the result contains `\`` (invalid Kafka topic char). Also register `A.Foo` and `B.Foo`; assert resolved equal.

Q38: `OutboxProcessorOptions.ComputeNextRetryAt` produces a *past* timestamp when `MaxRetryDelay` is configured to a negative `TimeSpan`, putting the failed message into an immediate-retry loop with no backoff.
   Evidence: `src/NetEvolve.Pulse/Outbox/OutboxProcessorOptions.cs:166-198`. With `MaxRetryDelay = TimeSpan.FromMilliseconds(-1)`, the condition `baseDelayMs > MaxRetryDelay.TotalMilliseconds` (line 174) is `>= -1` for any non-negative base delay, so `baseDelayMs = -1`; then `baseDelay = TimeSpan.FromMilliseconds(-1)`; clamp at line 189 stays negative; `now.Add(clampedDelay)` is in the past. No `ArgumentOutOfRangeException` is thrown at options-binding time ŌĆö `MaxRetryDelay` has no validator. The processor then reads `m.NextRetryAt <= now` (EF query at `EntityFrameworkOutboxRepository.cs:79`, `:112`) and immediately re-picks the row every poll.
   Why it matters: An operator typo (`-5m` instead of `5m` in config) silently disables backoff and creates a tight retry storm against the downstream broker, exactly the symptom backoff is supposed to prevent. Distinct from Q03 (which is about `DateTimeOffset.UtcNow` bypass of `TimeProvider`): Q38 is about arithmetic validity. Also note `BaseRetryDelay = TimeSpan.Zero` plus `AddJitter = false` produces `now.Add(0)` = retry-immediately even with positive `MaxRetryDelay`.
   Test idea: `var opts = new OutboxProcessorOptions { MaxRetryDelay = TimeSpan.FromSeconds(-30), EnableExponentialBackoff = true }; var next = opts.ComputeNextRetryAt(DateTimeOffset.UtcNow, retryCount: 3);` Assert `next < DateTimeOffset.UtcNow`. Wire this against a fake repository and assert the same message ID is re-fetched on three consecutive polls without elapsed wall-clock delay.

Q39: The default `IMessageTransport.SendBatchAsync` interface implementation calls `SendAsync` in parallel and is not atomic, yet `OutboxProcessorHostedService.ProcessBatchSendAsync` treats batch-send failure as "all messages failed, retry all" ŌĆö producing duplicate delivery for messages that succeeded before the failure.
   Evidence: `src/NetEvolve.Pulse.Extensibility/Outbox/IMessageTransport.cs:80-90` uses `Parallel.ForEachAsync` over `SendAsync`. The XML doc admits "may deliver a subset if an exception occurs mid-batch." `src/NetEvolve.Pulse/Outbox/OutboxProcessorHostedService.cs:428-500` catches *any* exception from `SendBatchAsync` and marks every message in the batch as failed/dead-letter, ignoring which ones actually succeeded. The comment at line 460-463 acknowledges this ("falling back to individual processing would re-send already-delivered messages") but the chosen behavior (mark all failed) achieves the same duplicate effect via retry.
   Why it matters: Transports that don't override `SendBatchAsync` (e.g., `NullMessageTransport`, custom user implementations, possibly Dapr/RabbitMQ unless their override is truly atomic ŌĆö see Q07 for ASB confirmed non-atomic when batching disabled) will deliver messages twice whenever any single send fails. The contract documentation says implementations "SHOULD provide at-least-once delivery" and "MUST be idempotent," but the framework itself amplifies duplicates instead of minimizing them. Distinct from Q07 (which is specifically about Azure Service Bus when `EnableBatching=false`): Q39 is about the *default interface implementation* applied to every transport that doesn't override.
   Test idea: Implement `IMessageTransport.SendAsync` that succeeds for message 0 and 1 but throws on message 2. Enable `EnableBatchSending = true`. Publish 3 events, run one processor cycle. Assert that on the next cycle, all 3 messages are retried (count delivery attempts via a thread-safe counter ŌĆö should be 5 total: 2 successes + 1 failure + 3 retries = at least 5).

Q40: `SystemTextJsonPayloadSerializer` uses `JsonSerializerOptions.Default` when no options are configured and registers no `TypeInfoResolver` for polymorphism ŌĆö payloads typed as base classes/interfaces serialize only declared members, losing the runtime subtype data on round-trip.
   Evidence: `src/NetEvolve.Pulse/Serialization/SystemTextJsonPayloadSerializer.cs:28` (`_options = options.Value ?? JsonSerializerOptions.Default`). No `[JsonPolymorphic]`/`[JsonDerivedType]` attributes on `OutboxMessage`, `IEvent`, or the payload contract. `Serialize(value, messageType)` at line 35 uses the runtime type passed by `OutboxEventStore.cs:84` (`message.GetType()`), which would normally include derived fields ŌĆö BUT the consumer side reads `EventType` (a `Type`) and deserializes; if the payload itself has polymorphic *fields* (e.g., `IPayload Data { get; }`), STJ defaults will write the declared interface members only.
   Why it matters: Domain events frequently wrap heterogeneous payloads (`OrderEvent { ICustomer Customer }`, discriminated unions, F# records). The serializer happily round-trips these in unit tests where the static type matches the dynamic type but loses data on real events. Worse, deserialization will silently succeed and produce `null` for missing properties, so the consumer can't even detect the loss. Distinct from Q19 (source-gen multi-targeting): Q40 is about the runtime serializer contract.
   Test idea: Define `record Wrapper(IPayload Data) : IEvent` and `record ConcretePayload(string Secret) : IPayload`. Serialize `new Wrapper(new ConcretePayload("X"))` via `SystemTextJsonPayloadSerializer`. Deserialize as `Wrapper`. Assert `result.Data` is `null` or empty ŌĆö proving the polymorphic information is lost.

   Evidence: `src/NetEvolve.Pulse.Extensibility/Outbox/OutboxMessage.cs:27-106` defines only `CorrelationId` and `CausationId` string columns.    Why it matters: Operators correlating a request across publisher ŌåÆ outbox ŌåÆ consumer will see disconnected traces, defeating distributed-tracing observability.    Test idea: Start an activity, call `mediator.PublishAsync(evt)`, run the outbox processor in the same test fixture. Q42: `IdempotencyCommandInterceptor` does not validate that `IdempotencyKey.Length <= 500`, so the handler executes successfully and then `IIdempotencyKeyRepository.StoreAsync` fails with a DB constraint error ŌĆö leaving the side effect in place but the idempotency record absent, so the next retry runs the handler again.
   Evidence: `src/NetEvolve.Pulse/Interceptors/IdempotencyCommandInterceptor.cs:68-79`: the key is used as-is. `src/NetEvolve.Pulse.Extensibility/Idempotency/IdempotencyKeySchema.cs:50-56` declares `MaxLengths.IdempotencyKey = 500`.    Why it matters: Pairs with Q04's check-then-act race. Q45: `EntityFrameworkOutboxRepository.GetPendingAsync` does not apply `FOR UPDATE SKIP LOCKED` (or any pessimistic hint), so two `OutboxProcessorHostedService` instances polling the same EF Core-backed database double-process the same message.
   Evidence: `src/NetEvolve.Pulse.EntityFramework/Outbox/EntityFrameworkOutboxRepository.cs:77-87` and `:108-120` use plain LINQ (`Where`/`OrderBy`/`Take`) and forward to `_executor.FetchAndMarkAsync`. Q46: `EndpointRouteBuilderExtensions.MapCommand` / `MapQuery` do not differentiate client-cancellation (`HttpContext.RequestAborted` ŌåÆ `OperationCanceledException`) from internal cancellation/timeout, so client disconnects bubble up as ASP.NET Core's default 500 Internal Server Error in the response logs instead of the intended 499/204 semantics.

   Why it matters: Distinguishing client abort from server failure is a standard observability requirement. Q47: `RateLimitedEventDispatcher.DispatchAsync` swallows handler exceptions whose `CancellationToken` was *already* cancelled (e.g., a handler that throws after partial work past its own internal timeout), while `PrioritizedEventDispatcher.DispatchAsync` always captures them ŌĆö inconsistent behavior across dispatchers.
